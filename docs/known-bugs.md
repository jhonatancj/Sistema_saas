# Known bugs — patrones recurrentes y su causa raíz

Referencia de troubleshooting: síntoma → causa → fix. No es un changelog — si un
bug se corrige de raíz y el patrón ya no puede recurrir, se borra de acá. Si un
riesgo estructural sigue vivo, vive en `CLAUDE.md` → "Riesgos técnicos conocidos"
o en `CURRENT_STATE.md` → "Bugs abiertos", no en este archivo.

| Síntoma | Causa | Fix |
|---|---|---|
| Actualicé un form en `public` (`json_form`/`grid_query`/campos nuevos) y no se ve el cambio al probar en un tenant real (`tenant_demo`/`tenant_acme`) | `copyMissingFormsToTenant()`/el sync es **copy-if-missing** a propósito (ADR-013, para no pisar personalizaciones del tenant) — un form ya sincronizado antes del cambio se queda con su copia vieja para siempre, nada la actualiza sola. Pasó 3 veces en la misma sesión (grid_query de `producto_barrio`, campos DIAN de `proveedores`, layout de `venta_barrio`) | Empujar el mismo cambio a mano a la copia del tenant (`AdminFormsService.updateTenantForm(tenantSlug, slug, {...})` con el `json_form`/`gridQuery` ya corregido de `public`) — **nunca** copiar un `grid_query` verbatim entre schemas sin adaptar el nombre de schema (ver fila aparte). Si un form tiene SP a mano (`recreate_sp=false`), pasar `recreateSp:false` explícito en esa llamada para no pisarlo (ver ADR-020) |
| `relation "undefined.forms"` | Super admin accediendo un endpoint tenant-only | Aplicar `TenantGuard` al controller/ruta |
| `null value in "role_code"` | Query devuelve `code` pero el INSERT espera `role_code` | Usar `r.code AS role_code` en el SELECT |
| `tbl_X sin columnas` tras generar | `processForm` llamado con `json_form = null` | Validar que `json_form.root` exista antes de generar DDL |
| `has_table = false` tras `processForm` exitoso | El UPDATE de `processForm` no seteaba `has_table` | El UPDATE debe incluir `has_table = TRUE` junto con `has_sp = TRUE` |
| Menú del tenant vacío | `isAdmin()` llamado sin `()` — es un `computed` signal, no un booleano | `this.isAdmin()` |
| CORS bloqueado en dev | Subdominio no matchea la regex de `main.ts` | `main.ts` usa `/\.localhost(:\d+)?$/` |
| `PayloadTooLargeError` al guardar un campo `image` | Límite default de `express.json()` (100kb); el campo manda base64 completo | `main.ts` usa `app.use(json({ limit: '10mb' }))` — si persiste, la imagen supera ese límite |
| `500` al guardar `image` con el body-limit ya alto | Columna generada como `VARCHAR(500)` en vez de `TEXT` (tablas creadas antes del fix, ADR-006) | `ALTER TABLE {schema}.tbl_X ALTER COLUMN imagen TYPE TEXT;` manual — `processForm` no altera tipos de columnas existentes |
| Grid muestra el form anterior al navegar por el sidebar | Angular reutiliza la instancia del componente entre rutas `/app/m/:moduleCode/:formSlug` con params distintos — `ngOnInit` no se dispara de nuevo | Suscribirse a `route.paramMap` (Observable) en el constructor, nunca leer `route.snapshot` una sola vez |
| `AG Grid: error #239` en consola | AG-Grid v33+ activa la Theming API nueva por default; el proyecto usa el sistema viejo de CSS | `provideGlobalGridOptions({ theme: 'legacy', ... })` en `ag-grid.init.ts` |
| `relation "tbl_X" already exists` al abrir un form | `has_table=false` en metadata pero la tabla ya existe (corrida previa de `processForm` no transaccional, ver ADR-003) | `buildTableDDL` usa `CREATE TABLE IF NOT EXISTS` — se auto-corrige al reprocesar |
| Botón "Guardar" del builder no hace nada en la pestaña Grid/SQL, sin error visible | El componente que escucha el trigger de guardado (`<d-builder>`) se desmontaba con `@if` según la pestaña activa | Mantener `<d-builder>` siempre montado, ocultar con CSS (`display:none`), nunca con `@if` |
| Un form nuevo en modo "Por tenant" se crea en el catálogo público en vez del tenant elegido | Faltaba un endpoint de creación scopeada a un tenant específico | `POST /admin/forms/tenant/:tenantSlug` |
| `column "deleted_at" does not exist` (u otra columna) en un tenant específico | Ese tenant fue creado con una versión más vieja de `create_tenant_schema()` que la del archivo actual | Correr las migraciones pendientes de `Back/database/migrations/` contra ese tenant (`pnpm db:migrate --tenant=<schema>`) |
| Un tenant nuevo queda con solo 6 tablas (`users/roles/permissions/user_roles/role_permissions/refresh_tokens`) en vez de 11 | La función `create_tenant_schema()` viva en Postgres estaba desactualizada respecto al archivo `04_create_tenant.sql` — un bug de escaping SQL en el archivo (comillas sin duplicar dentro de un `EXECUTE format('...')`) impedía reaplicarlo | Corregido el escaping; si se sospecha drift de nuevo, re-aplicar `04_create_tenant.sql` (`CREATE OR REPLACE FUNCTION`, seguro, no toca tenants ya creados) y comparar `pg_get_functiondef` contra el archivo |
| `too few arguments for format()` al crear un tenant | Un bloque `EXECUTE format('...', args)` en `04_create_tenant.sql` con más placeholders `%I` que argumentos | Al editar cualquier bloque `format()` de ese archivo, contar `%I`/`%L`/`%s` uno por uno contra la lista de argumentos antes de aplicar |
| Sincronizar módulos a un tenant duplica los formularios asignados en cada corrida | `{schema}.module_forms` no tenía constraint única — `ON CONFLICT DO NOTHING` sin un target real no evita nada | `UNIQUE (module_id, form_slug)` en `04_create_tenant.sql` (tenants nuevos) + migración `002_module_forms_unique.sql` (tenants existentes, con dedup previo) |
| Bind de formulario público ofrece tablas como `public.users`/`public.roles` como "tabla existente" | Existen tablas huérfanas en `public` (drift de un prototipo pre-multi-tenant) sin relación con el motor de formularios | `AdminFormsService.CORE_PUBLIC_TABLES` las excluye explícitamente de la lista de bind |
| Un form en el sidebar del super admin aparece con `name: null` | `{schema}.module_forms.form_slug` es un `VARCHAR` sin FK hacia `forms.slug` (ni en `public` ni en tenant) | Resuelto para el camino normal: `FormGeneratorService.deleteForm()` (usado por `DELETE /admin/forms/:slug` y `DELETE /admin/forms/tenant/:tenantSlug/:slug`, expuesto desde el builder) borra `module_forms` del slug dentro de la misma transacción que dropea tabla/SP. Sigue pudiendo pasar si alguien borra una fila de `{schema}.forms` a mano por SQL sin pasar por este servicio — no hay FK que lo impida a nivel de DB |
| **[RESUELTO DE RAÍZ]** Un modal se renderizaba sin overlay/posición fija, o un botón "+ Nuevo" salía con el estilo por defecto del navegador | `.backdrop`/`.modal`/`.field`/`.btn` no eran globales — cada componente las redeclaraba en su propio `.scss`; 4 componentes (`builder`, `admin-modules`, `form-detail`, `settings-modules`) tenían el HTML pero no el bloque de estilos | Centralizado en `styles/components/` (capa ITCSS nueva) — ver `docs/adr/010-shared-component-styles.md`. Ya no puede volver a pasar por "olvidar copiar el bloque" para estas clases específicas; sí puede pasar con una clase nueva que todavía no exista en esa capa |
| Un formulario nuevo tiene tabla+SP generados pero la grid no muestra ninguna columna (se ve vacío) | `FormDetailComponent.colDefs` arma las columnas solo desde `{schema}.forms.grid_config`, nunca desde `json_form` — y `FormGeneratorService.processForm()` no toca `grid_config`, solo lo hace `saveGridConfig()` (la pestaña "Grid" del builder) | Si el form se creó llamando a `processForm()` directo (por script/API, sin pasar por la pestaña "Grid" del builder), hay que llamar también a `saveGridConfig(slug, columns)` con una entrada por campo (`visible: true`) — un form nuevo necesita los dos pasos, no alcanza con generar tabla/SP. Ver `docs/adr/013-catalogo-modulos-multi-vertical.md` |
| **[RESUELTO DE RAÍZ]** Un módulo recién sincronizado a un tenant tiene su form (`{schema}.forms`) pero la tabla real (`tbl_*`) no existe (`has_table=false`) | `copyMissingFormsToTenant()` solo copiaba la *definición* (`json_form`) — nunca llamaba a `processForm()` para generar la tabla/SP real de ese tenant | `ModulesService.syncPublicModulesToTenant()` corre `ensureFormsGenerated(schema, slugs)` inmediatamente después de copiar las definiciones — genérico para cualquier form recién asignado, no solo Categorías/Unidades (que fue donde se encontró el bug). Ver `docs/adr/015-catalogo-rubro-categorias-unidades.md` |
| **[RESUELTO DE RAÍZ]** Un campo numérico que debería coincidir (ej. "Rubro" del tenant) se muestra como "sin asignar"/vacío en el frontend, aunque el dato SÍ está seteado en la DB | `pg` (node-postgres) devuelve columnas `BIGINT` (OID 20 — el tipo de `id` en toda tabla `GENERATED ALWAYS AS IDENTITY`) como **string** por default en una query cruda (`pool.query`), pero el mismo tipo de dato sale como **number** cuando viaja a través de `to_jsonb()`/JSONB dentro de un SP del motor de formularios (`execute()`). Comparar ambos con `===` (ej. `tenants.rubro_id` vs. `tbl_rubro.id`) nunca matcheaba | `types.setTypeParser(20, parseInt)` registrado una sola vez en `Back/api/src/database/database.module.ts` — normaliza BIGINT a `number` en **toda** la app, para cualquier columna, no solo `rubro_id`. Si aparece un bug similar (una comparación de id que "debería" ser igual pero no lo es), verificar primero si alguno de los dos lados vino de una query cruda vs. de un JSONB del motor |
| **[RESUELTO DE RAÍZ]** Sincronizar un form a un tenant falla con `relation "tbl_x" does not exist` (ej. `empleados` antes que `sucursales`, o `venta_barrio` antes que `clientes`/`empleados`/`sucursales`/`producto_barrio`) | `ensureFormsGenerated()` itera `SELECT DISTINCT form_slug` sin orden garantizado por Postgres; `buildTableDDL()` genera la FK **inline en el mismo `CREATE TABLE`**, así que si la tabla referenciada no existe todavía el statement entero falla y aborta el resto del loop | `ModulesService.sortSlugsByDependencies()` — topological sort real sobre las dependencias `relation` de cada `json_form` (campos ocultos + columnas de `line-items`), reusando `FormGeneratorService.extractFields()`/`findLineItemsNode()`. `ensureFormsGenerated()` ordena antes de procesar. Ver `docs/adr/022-orden-dependencias-sync-tenant.md` |
| Un `grid_query` copiado de `public` a un tenant (a mano, no vía `copyMissingFormsToTenant` que no toca `grid_query` de un form ya existente) muestra 0 filas o datos de otro tenant | El texto del `grid_query` traía `public.tbl_x` horneado — mismo gotcha ya conocido para SPs a mano (`current_schema()`/schema horneado), pero en un `grid_query` en vez de una función | Nunca copiar un `grid_query` verbatim entre schemas — reconstruirlo con el nombre de schema del tenant destino, igual criterio que un SP a mano (ver fila de abajo sobre `current_schema()`) |
| Un select dependiente de `dforms` (`optionsParams`) nunca refiltra sus opciones pese a estar bien declarado en el `json_form` | `RemoteFormOptionsService.loadOptions(endpointId, params)` ignoraba el segundo argumento (`_params`) — dforms sí llama `loadOptions` con los params correctos, pero la implementación de la app nunca los reenviaba al backend | `loadOptions()` arma `filter.filters` (`{field, operator:'equals', value}`) por cada entrada de `params` y lo manda en el body de `POST /forms/:slug/execute` — `FormExecutorService.selectPaged()` ya soportaba filtrar así, validado contra `information_schema` |
| Un tenant nuevo no tiene una columna que sí está en `04_create_tenant.sql` (ej. `recreate_sp` en `forms`) | Editar el archivo `.sql` no alcanza — `create_tenant_schema()` ya vive como función en Postgres; un tenant nuevo llama a esa función, no relee el archivo | Reaplicar el archivo completo (`docker exec -i saas_postgres psql ... < Back/database/04_create_tenant.sql`, es un `CREATE OR REPLACE FUNCTION`, seguro, no toca tenants ya creados) antes de crear el tenant de prueba — mismo gotcha ya conocido para otras funciones (ver known-bugs.md, fila de `too few arguments for format()`) |
| Un SP a mano que consulta `current_schema()` para saber "en qué tenant estoy" nunca detecta el tenant correcto (siempre se comporta como si fuera `public`) | Esta app nunca hace `SET search_path` — toda query califica el schema explícitamente (`${schema}.tabla`). `current_schema()` refleja el `search_path` de la conexión, no el schema de la tabla que se está tocando ni el del SP invocado | El SP a mano debe recibir el nombre de schema como texto **literal horneado en su propio DDL de creación** (mismo criterio que `FormGeneratorService.buildSpDDL()` ya usa para los SPs autogenerados). Si un SP así se sincroniza a un tenant, hay que regenerar su texto con el schema de ese tenant, nunca copiarlo tal cual. Ver `docs/adr/017-tabla-detalle-line-items.md` |
| Al reabrir un registro guardado de un form con `line-items`, el campo de líneas aparece vacío (no muestra las líneas ya guardadas) | `<d-form-render>` solo se monta vía `@if (modalMode())` en `FormDetailComponent` — si `modalMode` se setea a `'edit'` ANTES de que resuelva el `SELECT_BY_ID` que trae el `detalle` anidado, el componente monta con la fila plana (sin líneas) y una actualización posterior de `[(submission)]` no vuelve a poblar un campo `line-items` ya montado | `openEdit()` espera la respuesta de `SELECT_BY_ID` (con el `detalle` ya anidado) antes de setear `modalMode` — el modal se monta una sola vez, ya con los datos completos, nunca dos veces |
| `pnpm install`/`pnpm build` falla en Docker (o en una máquina/CI nueva) con `[ERR_PNPM_IGNORED_BUILDS]` (`bcrypt` no compila) | pnpm 11 bloquea scripts `postinstall` de paquetes con binarios nativos por default (protección de supply-chain) — `Back/api/pnpm-workspace.yaml` ya tenía un `allowBuilds` con `bcrypt: false` y un placeholder sin completar (`'set this to true or false'` en `@scarf/scarf`), configuración a medio terminar de antes | `allowBuilds: { bcrypt: true, '@scarf/scarf': false, unrs-resolver: false }` en `pnpm-workspace.yaml` — **ojo**: el campo `"pnpm"` de `package.json` (`onlyBuiltDependencies`) ya NO lo lee pnpm 10+, solo `pnpm-workspace.yaml` (o `.npmrc`) |
| Build de Docker falla con `[ERR_PNPM_TARBALL_URL_MISMATCH]` o `[ERR_PNPM_MINIMUM_RELEASE_AGE_VIOLATION]` para `@jhonatancj/dforms` | (1) El token de GitHub Packages no llegó al build — pnpm no expande variables de entorno en credenciales que vengan de un `.npmrc` DE PROYECTO (protección contra un `.npmrc` committeado que filtre el secreto), hay que escribir el `_authToken` en un `.npmrc` de usuario (`~/.npmrc`), nunca en el del repo. (2) pnpm 11 bloquea instalar una versión publicada hace poco (`minimumReleaseAge`, default ~24h) — el `minimumReleaseAgeExclude` con la versión exacta no siempre lo evita en un store completamente limpio (ej. una imagen Docker nueva) | (1) En el `Dockerfile`, escribir el token vía build secret (`--mount=type=secret`) a `/root/.npmrc` justo antes de `pnpm install`, borrarlo después en el mismo `RUN`. (2) `minimumReleaseAge: 0` en `Front/pnpm-workspace.yaml` — más confiable que mantener el exclude-list versión por versión |
| **[RESUELTO DE RAÍZ]** Un SP a mano (`recreateSp:false`, ej. `sp_venta_barrio`) pierde su lógica de negocio (queda como el auto-generado básico, sin `line-items`/stock) sin ningún error visible | `recreateSp` no se persistía en `{schema}.forms` — era un parámetro transitorio de cada llamada a `processForm()`. Cualquier `updatePublicForm()`/`updateTenantForm()` posterior sobre ese slug que no pasara `recreateSp:false` explícitamente (default `true`) pisaba el SP a mano con el genérico. Agravado en el frontend: `BuilderComponent.resetAdvancedFields()` reseteaba el toggle "Regenerar SP" a `true` en cada carga del form (nunca leía el valor guardado) y `onExport()` siempre mandaba ese booleano explícito — con lo cual ni siquiera pasar `recreateSp:false` en el backend alcanzaba si alguien guardaba desde el builder UI | Columna nueva `recreate_sp BOOLEAN NOT NULL DEFAULT TRUE` en `{schema}.forms` (`public`, template de tenant y migración `004_forms_recreate_sp.sql`). `FormGeneratorService.processForm()`: `dto.recreateSp ?? (exists ? prev.recreate_sp : true)` — el valor guardado gana sobre el default. `AdminFormsService` ya no fuerza `?? true`, pasa `dto.recreateSp` tal cual. Frontend: `resetAdvancedFields()` ahora usa `form?.recreate_sp ?? true`. `venta_barrio`/`compra_barrio` (y todo SP a mano nuevo) se crean con `recreate_sp=false` desde el arranque |
| **[RESUELTO DE RAÍZ]** Un form recién sincronizado a un tenant (`copyMissingFormsToTenant`) siempre abre en modal aunque en `public` esté configurado como `inline` | El `INSERT ... SELECT` de `copyMissingFormsToTenant()` solo copiaba `(slug, name, json_form, grid_config, icon)` — `display_mode`/`modal_width` quedaban en su default de columna (`'modal'`/`NULL`) sin importar el valor real en `public.forms` | Agregadas `display_mode`/`modal_width` al `INSERT ... SELECT` de `copyMissingFormsToTenant()` (`Back/api/src/modules/modules/modules.service.ts`). Forms ya sincronizados antes del fix (`tenant_demo`/`tenant_acme`) corregidos a mano con un `UPDATE ... FROM public.forms` puntual |
| Un campo `number` oculto con `relation` (patrón `input-lupa`, ej. `empleados.sucursal_id`) queda como `NUMERIC(12,2)` **sin FK real** aunque el patrón ya esté fijado en ADR-019 | El nodo oculto del campo no tenía la propiedad `relation` en el `json_form` — sin ella, `toDbType()`/`buildTableDDL()` no tienen forma de saber que es una FK y generan una columna numérica suelta. Ya había pasado una vez con `cita`/`venta_barrio` (corregido en su momento) pero `empleados` mismo, el primer form que usó `input-lupa`, nunca se corrigió | `ALTER COLUMN ... TYPE BIGINT USING ...::BIGINT` + `ADD CONSTRAINT` idempotente (mismo patrón que `buildAlterTableDDL`), y agregar `relation: {form, keyValue}` al nodo oculto en el `json_form` para que el motor lo reconozca en cualquier reprocesamiento futuro. Si un campo `input-lupa` en cualquier form nuevo muestra el id pero no aparece como FK en `\d tbl_x`, revisar primero si al nodo oculto le falta `relation` |
